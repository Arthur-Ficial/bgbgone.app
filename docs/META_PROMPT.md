# Meta-prompt — stable GUI + App Store ready

> Drop this brief verbatim onto any AI building a desktop GUI app you want shipped to the Mac App Store. The prompt has two halves: STABILITY (the app never lies, never crashes, never hangs) and STORE-READY (Apple will accept it on first review, with Privacy + ToS + the lot).

---

## G1 — Real, stable UI

Every pixel on screen is derived from real state. No mock data, no placeholder art, no aspirational labels, no fake chrome, no fallbacks that hide failure.

- Use only stock OS primitives (SwiftUI / AppKit; no custom-painted OS chrome).
- Every view handles five states explicitly: `empty`, `loading`, `partial`, `success`, `error`.
- Every async path has timeout, cancellation, and user-visible progress.
- Every user input is validated at the boundary; internal code trusts internal code.
- Errors are surfaced verbatim with an actionable next step. No silent catches. No `try?` swallowing real failure paths.
- Strict concurrency on, models `Sendable`, `async`/`await` + `AsyncStream` only (no Combine, no completion handlers).
- ViewModels are `@Observable`. State machines are pure and testable.
- Quality gate: `make test && make build && make app && make run`, plus AI-driven screenshot verification of every state against the design spec.

**Tests pass ≠ UI works.** Look at the pixels. Build the bundle, launch it, screenshot every state, read the screenshot back, compare to the design spec.

## G2 — App Store-ready, first try

The repo ships everything Apple asks for.

### `Info.plist`
- `CFBundleIdentifier`, `CFBundleVersion`, `CFBundleShortVersionString` (driven from `.version`).
- `LSApplicationCategoryType` — pick the right App Store category.
- `NSHumanReadableCopyright`.
- `ITSAppUsesNonExemptEncryption` = `NO` (or proper export-compliance docs if `YES`).
- `LSMinimumSystemVersion`.
- `CFBundleIconFile`.

### Icon
`Assets.xcassets/AppIcon.appiconset` (or `AppIcon.icns`) with **all** required sizes (16, 32, 64, 128, 256, 512, 1024 at @1x + @2x).

### Privacy manifest
`PrivacyInfo.xcprivacy` declaring:
- `NSPrivacyTracking` = `false`.
- `NSPrivacyTrackingDomains` = `[]`.
- `NSPrivacyCollectedDataTypes` = `[]` (or the truth).
- `NSPrivacyAccessedAPITypes` — the exact list of required-reason APIs the app touches (file timestamps, user defaults, disk space, system boot time, file metadata). Declare empty if none.

### Entitlements
- App Sandbox `YES`, hardened runtime `YES`.
- Only the entitlements the app actually uses (`com.apple.security.files.user-selected.read-write`, `com.apple.security.files.bookmarks.app-scope` if persisting drops, etc.).
- No network entitlement unless the app uses the network.

### Public docs
- `PRIVACY.md` and `TERMS.md` checked into the repo, mirrored to a public URL (GitHub Pages or your domain), and linked from the App Store Connect listing.
- Privacy policy enumerates: what is collected (probably nothing — say so), where it goes (nowhere — say so), retention (n/a), third parties (none), contact email, effective date.

### App Store Connect metadata
`appstore/` directory:
- `name`, `subtitle`, `promotional_text`, `description`, `keywords`.
- `support_url`, `marketing_url`, `privacy_policy_url`.
- `category`, `age_rating`.
- `whats_new` (per version).
- Screenshots at the required Mac resolutions (`1280×800`, `1440×900`, `2560×1600`, `2880×1800`) generated by running the real app.

### Signing / notarisation / submission
- Code-signing with a real Developer ID Application certificate.
- Notarisation via `notarytool` against Apple's notary service, stapled via `stapler`.
- `make release` runs the full pipeline end-to-end and uploads to App Store Connect via `xcrun altool` or `transporter`.

### Hygiene
- No private APIs.
- No deprecated APIs.
- No third-party SDKs for problems the OS already solves.
- Crash reporting via the OS's own facility (no Crashlytics).
- Analytics: none, or opt-in with a checkbox the user must tick — never default-on.
- `README.md` documents install (Homebrew cask + Mac App Store), build, test, release.
- Every public type and method in `Sources/` has a doc comment explaining *why* it exists, not what it does.

## Execution rules

1. Do not invent fallbacks. If something is missing, fail loud with the real error.
2. Do not add a feature without a test. TDD — red, green, refactor.
3. Do not claim done without: (a) green tests, (b) clean build, (c) launched bundle, (d) screenshot of every state read back and visually verified against the design spec.
4. Documentation is part of "done", not a follow-up. Every file > 30 lines has a header comment explaining its role; every non-trivial function has a one-line `///` doc.
5. Commit after each green gate. Tag releases. Push to GitHub.

## Definition of done

A stranger can clone the repo, run `make release`, get a notarised `.zip` *and* an App Store-ready build, and Apple will approve it on first submission.
